# 跳广告 App 8 周开发计划（优化版）

## 1. 计划目标

本计划以 **Kotlin + Jetpack Compose + AccessibilityService** 为核心，按照“**先验证核心能力，再抽象规则系统，再完善体验，最后做稳定性与发布**”的顺序推进，目标是在 8 周内完成一个可用、可测、可维护的 Android 跳过启动页广告工具。

计划优化原则：

- 先做最小闭环，避免一开始就引入数据库、复杂 UI、远程同步等高成本模块。
- 所有阶段必须有明确验收标准，避免“功能看起来差不多”但无法判断是否完成。
- 优先控制误触、漏触、耗电、后台稳定性，这四项直接决定产品是否可用。
- 规则系统和 UI 必须服务于核心引擎，而不是反过来拖慢引擎迭代。
- 所有新增能力默认要求可观测，至少有日志、耗时统计或失败原因记录。

## 3. 阶段拆解

## 第一阶段：核心引擎验证（第 1-2 周）

### 阶段目标
实现最小可用的无障碍服务，能够稳定识别并点击主流 App 的“跳过”按钮。

### 本阶段只做

- AccessibilityService 基础接入
- 节点树遍历与规则匹配
- 点击执行链路
- 少量硬编码规则实机验证
- 完整日志与耗时统计

### 本阶段明确不做

- 不做 UI
- 不接 Room / DataStore
- 不做规则编辑器
- 不做前台保活
- 不做云端同步与高级识别

### 任务清单

| 任务项 | 具体内容 | 验收标准 | 关键技术点 |
| --- | --- | --- | --- |
| 环境搭建 | 初始化 Kotlin + Compose 项目；配置 accessibility_service_config.xml | 服务可在系统设置中正常开启/关闭 | canRetrieveWindowContent=true |
| 节点遍历封装 | 实现 BFS 遍历节点；支持 text/id/desc 匹配 | 单次全树遍历耗时 <= 50ms | 迭代队列、空节点安全、节点去重 |
| 窗口事件监听 | 监听 TYPE_WINDOW_STATE_CHANGED / TYPE_WINDOW_CONTENT_CHANGED | 仅在目标 App 页面触发识别 | 事件节流、包名缓存、重复事件抑制 |
| 模拟点击 | 支持 ACTION_CLICK 与 dispatchGesture 双模式 | 不可点击节点可回溯到可点击祖先 | 点击链路兜底、手势注入 fallback |
| 基础规则集 | 内置 5-10 个主流 App 规则做实机测试 | 冷启动广告跳过成功率 >= 80% | 规则命中日志、耗时记录 |
| 故障日志 | 输出命中规则、节点路径、失败原因、点击耗时 | 能通过 Logcat 快速定位失败原因 | 分级日志、失败归因 |

### 周交付

- 第 1 周：服务可启用，能监听目标窗口并遍历节点树。
- 第 2 周：完成点击闭环，内置规则可在多款 App 上实机验证。

### 阶段验收指标

- 5-10 个目标 App 中，至少 80% 可成功跳过广告。
- 单次识别 + 匹配 + 点击平均耗时 <= 100ms。
- 出现失败时，日志可定位是“未命中规则、节点不可点击、事件未触发”中的哪一类。

### 主要风险与控制

- 风险：不同 App 节点结构变化快。
- 控制：规则同时支持 text、viewId、contentDescription、多条件组合。

- 风险：重复事件导致误触或多次点击。
- 控制：增加时间窗节流和页面签名缓存。

## 第二阶段：规则系统与数据层（第 3-4 周）

### 阶段目标
将硬编码规则解耦为可配置规则引擎，并支持本地持久化与热更新。

### 任务清单

| 任务项 | 具体内容 | 验收标准 | 关键技术点 |
| --- | --- | --- | --- |
| 规则模型设计 | 定义 JSON Schema：包名、触发条件、动作、超时、优先级、启用状态 | 规则文件可通过 JSON 校验 | Kotlin Serialization 或 Moshi |
| 规则管理器 | 实现加载、匹配、缓存、热更新 | 修改规则后无需重启服务即可生效 | 内存缓存、版本号、增量刷新 |
| 本地存储 | Room 存用户规则；DataStore 存全局配置 | CRUD 无主线程阻塞 | Flow、IO 调度、Repository 分层 |
| 导入导出 | 支持从剪贴板/文件导入 JSON，支持文件导出分享 | 非法规则有明确错误提示 | SAF、异常提示、格式校验 |
| 内置规则集 | 整理 50+ 常见 App 规则并放入 assets | 开箱即用覆盖主流场景 | assets 预置、首启导入 |
| 社区兼容 | 尽量兼容 GKD 等规则结构 | 可导入常见社区规则并转换 | 字段映射、兼容层 |

### 周交付

- 第 3 周：完成规则模型、规则匹配器、规则文件加载。
- 第 4 周：完成 Room + DataStore、导入导出、内置规则资产化。

### 阶段验收指标

- 修改本地规则后，服务可在不中断的前提下重新加载。
- 规则 CRUD 平均响应时间 <= 100ms。
- 导入非法 JSON 时，有可理解的错误信息，不导致服务异常。

### 工程要求

- 引擎层与数据层解耦，规则匹配不直接依赖 UI。
- 规则模型要预留扩展字段，避免后续新增动作时频繁改表。
- 所有磁盘 IO 与 JSON 解析都必须放在后台线程。

---

## 第三阶段：UI 与用户体验（第 5-6 周）

### 阶段目标
提供完整的用户交互界面，降低使用门槛，并合规引导用户完成权限授予。

### 任务清单

| 任务项 | 具体内容 | 验收标准 | 关键技术点 |
| --- | --- | --- | --- |
| 主页仪表盘 | 展示服务状态、今日跳过次数、最近触发记录 | 数据实时更新 | Compose + StateFlow |
| 权限引导页 | 分步引导开启无障碍、关闭电池优化、允许后台运行 | 每步都有跳转设置按钮 | Intent 精确跳转、状态检测 |
| 规则编辑器 | 可视化编辑规则，支持实时预览匹配条件 | 保存后立即生效 | 表单校验、字段联动 |
| 日志查看器 | 支持筛选、复制、清空、反馈 | 日志敏感信息自动脱敏 | LazyColumn、分页、文本脱敏 |
| 隐私声明 | 首次启动必须阅读并同意 | 未同意则不能启用核心功能 | DataStore 记录同意状态 |

### 周交付

- 第 5 周：完成主页、服务状态、权限引导。
- 第 6 周：完成规则编辑器、日志查看器、隐私声明流程。

### 阶段验收指标

- 新用户可在 3 分钟内完成权限配置并开始使用。
- 修改单条规则后，能够即时生效并在日志中看到命中结果。
- 日志列表在大量记录下仍保持流畅，无明显掉帧。

### 设计要求

- 页面结构保持轻量，先满足“可用”和“好理解”，不要过度追求复杂视觉效果。
- 所有开关项必须说明对功耗或稳定性的影响。
- 敏感信息默认脱敏，包括包名以外的可识别文本内容。

---

## 第四阶段：稳定性与功耗优化（第 7 周）

### 阶段目标
确保服务具备长期后台运行能力，并将额外功耗控制在合理范围内。

### 任务清单

| 任务项 | 具体内容 | 验收标准 | 关键技术点 |
| --- | --- | --- | --- |
| 前台服务 | 创建常驻通知并绑定核心服务生命周期 | 服务被系统回收概率显著降低 | 前台服务、通知渠道 |
| 厂商适配 | 适配 MIUI / ColorOS / OriginOS 等常见 ROM 引导页 | 主流国产机型后台存活率 >= 90% | 厂商设置 Intent 兼容 |
| 性能分析 | 使用 Systrace / Perfetto 分析回调耗时 | P99 延迟 < 30ms | 热路径优化、减少主线程工作 |
| 异常熔断 | 连续失败 N 次自动暂停该 App 识别 | 防止死循环误触和持续耗电 | 滑动窗口计数器、自动恢复 |
| 省电模式 | 降低检测频率、关闭高级识别、按需停用前台能力 | 额外耗电 < 0.5%/h | 配置驱动的行为降级 |

### 周交付

- 第 7 周上半周：完成前台服务、厂商适配引导。
- 第 7 周下半周：完成性能分析、异常熔断、省电模式。

### 阶段验收指标

- 主流测试设备连续运行 24 小时，核心服务不异常退出。
- 高频广告场景下无明显误触，无持续点击死循环。
- 开启省电模式后，功能可用性下降可控且功耗明显降低。

## 第五阶段：测试与发布准备（第 8 周）

### 阶段目标
完成多设备验证、隐私合规自查、发布文档与签名产物准备。

### 任务清单

| 任务项 | 具体内容 | 验收标准 | 关键技术点 |
| --- | --- | --- | --- |
| 真机矩阵测试 | 覆盖 Android 9-15 及主流厂商 ROM | 各机型核心功能正常 | 云测平台 / 志愿者内测 |
| 边界场景测试 | 锁屏、来电、分屏、横竖屏切换、快速切 App | 无崩溃、无资源泄漏、无误触 | 自动化 UI 辅助 + 人工回归 |
| 合规自查 | 隐私政策、权限说明、功能用途说明 | 符合国内 App 合规基线 | 权限最小化、隐私文案审查 |
| 发布文档 | README、使用说明、规则贡献指南、FAQ | 文档清晰、截图完整 | GitHub 文档、版本说明 |
| 正式签名 | 配置签名密钥、R8/ProGuard、发布构建 | APK 体积 < 5MB，发布流程稳定 | 签名外部化、混淆规则 |

### 周交付

- 完成测试清单与回归报告。
- 产出可发布构建与配套文档。

### 阶段验收指标

- 所有 P0 / P1 问题关闭后再发布。
- 发布包可稳定安装、授权、运行、跳过广告。
- README 能覆盖安装、授权、规则导入、常见问题处理。

## 4. 推荐技术架构

### 模块分层建议

- app：Compose UI、导航、依赖注入入口
- core-accessibility：无障碍事件监听、节点遍历、点击执行
- core-rule：规则模型、规则匹配器、规则转换器
- data：Room、DataStore、Repository
- feature-settings：权限引导、全局配置
- feature-rules：规则列表、规则编辑、导入导出
- feature-logs：日志查看与反馈

如果当前项目规模较小，前 4 周可以先保留单模块结构，但代码目录上仍建议按 domain 分包，避免后期重构成本过高。

### 推荐分包

- accessibility/
- rule/
- data/
- log/
- ui/
- settings/
- common/

## 5. 核心指标看板

建议从第一阶段开始持续记录以下指标：

- 跳过成功率
- 误触率
- 平均识别耗时
- P95 / P99 回调耗时
- 单日触发次数
- 连续失败次数
- 后台存活时长
- 每小时额外耗电

这些指标后续会直接决定是否需要继续优化规则、性能或保活方案。

## 6. 优先级定义

### P0：必须完成

- 无障碍服务可稳定运行
- 可识别并点击主流 App 的跳过按钮
- 基础规则系统可配置
- 核心日志可定位问题

### P1：强烈建议完成

- 用户规则编辑与导入导出
- 权限引导与隐私声明
- 前台服务与异常熔断
- 多机型兼容测试

### P2：可视时间投入决定

- 社区规则兼容增强
- 更复杂的条件匹配
- 高级识别模式
- 更完整的厂商适配库

---

## 7. 关键风险清单

| 风险 | 影响 | 控制策略 |
| --- | --- | --- |
| 广告页结构频繁变化 | 跳过失败率上升 | 规则多条件匹配 + 热更新机制 |
| 系统限制后台运行 | 服务失效 | 前台服务 + 厂商适配引导 |
| 误触主内容按钮 | 用户体验严重受损 | 优先级规则、点击前二次校验、失败熔断 |
| 日志过多影响性能 | 拖慢服务 | 分级日志、调试模式开关 |
| 规则复杂度失控 | 维护成本上升 | Schema 固化、字段治理、版本管理 |

---

## 8. 建议开发节奏

- 每周开始前确认该周验收指标，不做与本周目标无关的扩展。
- 每周至少进行一次 3-5 台设备的小规模回归。
- 所有“看起来不错”的改动，如果不能提升成功率、稳定性或可维护性，应延后处理。
- 先保证 10 个 App 稳定，再追求 50+ 规则覆盖。

---

## 9. 最终交付定义

8 周结束时，项目应至少满足以下条件：

- 用户可以安装并完成权限配置。
- 服务可以在多款常见 App 中自动跳过启动广告。
- 规则支持本地编辑、导入、导出与持久化。
- 日志、隐私声明、基本设置、错误保护机制完整。
- 项目具备测试记录、发布文档和正式签名产物。

如果时间不足，优先保证 **核心引擎稳定 + 规则系统可维护 + 权限引导清晰**，其余体验增强项可延后到后续版本。
